## 前置信息收集

执行前确认用户已提供以下信息：

| 字段 | 是否必填 | 说明 |
|------|----------|------|
| 收件人（To） | ✅ 必填 | 支持多个，逗号分隔 |
| 主题（Subject） | ✅ 必填 | 邮件标题 |
| 正文（Body） | ✅ 必填 | 邮件内容 |
| 抄送（CC） | ⬜ 可选 | 支持多个，逗号分隔 |
| 附件路径 | ⬜ 可选 | 需提供完整绝对路径，多个路径换行或逗号分隔 |

> ⚠️ **附件路径预验证**：若用户提供了附件路径，在进入执行流程前，
> 先逐一验证每条路径是否存在。若任意路径不存在，立即暂停并告知用户
> 具体哪条路径有误，不得进入后续流程。

## 执行流程

## Classic Outlook 快捷键策略

在已确认目标窗口和焦点正确时，优先使用以下确定性快捷键，减少 UIA 扫描和模型调用：

| 场景 | 快捷键 | 约束 |
|---|---|---|
| 新建邮件 | `Ctrl+N` | 必须先激活 Outlook 邮件主窗口 |
| 字段间前进 | `Tab` | 仅在写信窗口且当前焦点已知时使用 |
| 字段间后退 | `Shift+Tab` | 仅用于纠正一次已知的焦点偏移 |
| 发送邮件 | `Alt+S` | 外部副作用；必须完成用户确认后使用 |

快捷键应通过 `hotkey` / `press_key` 调用；连续的点击、输入、Enter、Tab 操作应合并为
一次 `run_actions`。如果窗口或焦点未知，先用 `list_windows` / `list_elements` 确认，
不要盲目重复快捷键。

### Phase 1 — 启动 Outlook 并切换至邮件视图

1. 调用 `app_launch` 启动或激活 Outlook 主窗口
2. 读取自动验证的 `[屏幕观察]` 结果，判断当前是否在邮件视图（Inbox/收件箱）：
  - **如果当前处于日历/联系人等非邮件视图**：需要找到并点击 `Mail` 切换回Inbox视图
  - 确认已处于收件箱视图后，优先调用 `hotkey(keys="Ctrl+N", window="<Outlook主窗口>")`
  - 调用一次 `list_windows` 查找并激活新建邮件窗口
  - 只有快捷键没有创建窗口时，才回退到 UIA 定位并点击 `New Email`

### Phase 2 — 一次性批量定位邮件编辑区控件

** 一次 UIA 扫描获取全部编辑控件坐标，后续填写操作均使用坐标驱动， 禁止逐个 `find_and_click`。**

扫描目标控件（`batch_locate_elements` 参数 `targets`）：

⚠️ **To/Cc 特殊处理**：`batch_locate_elements` 返回的是标签按钮坐标（如 `(786,594)`），实际的收件人输入区域在按钮右侧。

---

### Phase 3 — 按字段逐一填写邮件字段

> **每个字段填写完后，点击 `Enter` 进行确认**。
> 无字段的步骤可跳过（如无 Cc 则跳过 Cc 步骤直接到 Subject）。
> 使用 `Tab` 键进行字段切换
> 已通过 UIA 获得输入区域坐标后，将 click/type_text/Enter/Tab 合并到一次
> `run_actions`；不要为每个字符或每个字段重新调用模型。

#### 填写顺序

```
1. To（偏移点击 → type_text）
2. Cc（偏移点击 → type_text，无抄送跳过）
3. Subject（直接点击 → type_text）
4. Body（直接点击 → type_text）
```
---

### Phase 4 — 添加附件（仅用户提供附件路径时执行）

> 📌 **核心原则**：每次只添加一个附件，通过"粘贴完整路径"方式精确定位文件，
> 不依赖目录浏览。多附件时循环执行，每轮独立完成。


### Phase 5 — 发送邮件

完成外部副作用确认后，优先在写信窗口调用 `hotkey(keys="Alt+S")`。如果快捷键未生效，
再回退到 UIA 定位并点击 `Send` 按钮。

---

### Phase 6 — 发送验证

```
analyze_screen(prompt="Outlook 当前是否回到了收件箱/主窗口？新邮件窗口是否已关闭？只回答是或否。")
```

**验证逻辑：**

```
第一次检测
├── 是 → ✅ 告知用户"邮件已成功发送"
└── 否 → sleep(2) 后执行第二次检测
          ├── 是 → ✅ 告知用户"邮件已成功发送"
          └── 否 → 📸 截图上报，告知用户手动检查
```
